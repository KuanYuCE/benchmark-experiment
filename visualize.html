128-bit Euclidean 時間（3.48s）是 64-bit 的 3.6×，主因是每次 mod 從硬體指令（1.7 ns）變成軟體模擬（4.6 ns）。 Binary 的 128-bit 成本僅 3.3×，因為位移操作不受影響。

t_ratio（euc / bin，>1 表示 Binary 勝）

ops_ratio 全程穩定在 ~0.83， 代表 Euclidean 在所有 tier 都比 Binary 少做約 17% 的 iteration。 Binary 贏靠的不是「更少操作」，而是每次操作更便宜。

ops_ratio 趨勢（euc_ops / bin_ops）

ops_ratio 幾乎是水平線（0.830–0.861），說明兩個演算法的 iteration 比例是固定的， 與數字大小無關。時間上的勝負完全由每次操作的硬體成本決定。

除法原始成本 Micro-benchmark（ns/op）

這是 Binary 在 128-bit 翻盤的直接原因：軟體模擬除法比硬體除法貴 2.8×， 而 Binary GCD 的位移/減法操作成本完全不受影響。

🔍 意外發現：GMP 在 64/128-bit 反而比 hybrid 快。 GMP 的 mpz_gcd 內部已有高度最佳化的 1-limb 路徑（mpn_gcd_1）與 2-limb 路徑， 比我們手寫 dispatch + mpz_to_u128 / u128_to_mpz 轉換成本還低。 Hybrid 的價值在於避免進入 mpz_t 世界，而非在 GMP 內部插一層。

64/128-bit：hybrid 反而較慢（轉換 overhead 抵消了 fast path 節省的計算）。 256-bit 起兩者幾乎貼合，因為 hybrid 直接 fall-through 到 mpz_gcd， 只多了 2 次 mpz_size() 比較的微小 overhead。

hybrid speedup（mpz_gcd 時間 / hybrid 時間，>1 表示 hybrid 勝）

64/128-bit 的 speedup 遠低於 1.0，說明 GMP 對小數字有更好的內部 fast path。 256-bit 以上 speedup 穩定在 1.0 ± 0.01（noise 等級），hybrid fall-through 的 overhead 幾乎為零。

mpz_gcd 隨位元寬的時間縮放（GMP Lehmer/HGCD 演算法行為）

256-bit → 2048-bit 的時間縮放相對平緩（非線性成長），反映 GMP Lehmer 演算法的高效率。 若用 naive Euclidean with mpz_mod，此處會呈現 O(n²) 的急劇增長。
